Actirova AI Tutor — Complete System Documentation

Repo: actinova-ai-tutor (Next.js + MongoDB + OpenAI + Paystack)

NOTE: The codebase/documentation sometimes references “Actinova” and “Actirova”. In practice, they refer to the same product.

============================================================
0. Overview / Goals
============================================================
Actirova AI Tutor is a full-stack AI learning platform built with Next.js (App Router), React, MongoDB, OpenAI, and Paystack. The platform provides:
- AI-generated structured courses (modules/lessons)
- Flashcards (SRS-aware)
- Practice quizzes
- Long-form report generation and editing/validation
- AI tutor chat for focused learning
- Personalization from onboarding and usage history
- Premium subscriptions (Paystack) and tier-based limits
- Career tools (resume/interview/skill-gap/network/portfolio)
- Library browsing, progress support, and share-link access
- Cron jobs for trending refresh, plan expiry enforcement, and content generation

2. High-Level Architecture
============================================================
2.1 Layering
- UI routes & components: src/app/** and src/app/components/**
- API endpoints: src/app/api/**
- Shared business logic: src/app/lib/**
- DB connection helpers/models: src/app/lib/dbConnect.js, src/app/lib/db.js, src/app/models/**
- Cron/background logic:
  - protected cron endpoints: src/app/api/cron/**
  - runner: scripts/run-crons.js

2.2 Request flow (typical protected mutation)
1) middleware.js (edge/Next middleware) checks path access
2) API route composes middleware:
   - withAuth: verify JWT & load user + ensure account active/locked
   - withCsrf: require valid CSRF token for unsafe methods
   - withAPIRateLimit: enforce plan tier quotas (no automatic usage increment)
   - withErrorHandling: standardized JSON errors
3) Handler executes OpenAI/MongoDB work
4) Handler stores results (library/tests/reports/etc.)
5) Handler increments usage explicitly via trackAPIUsage where required

5. Security Model
============================================================
5.1 middleware.js (path-level protection)
File: middleware.js (repo root)

What it does:
- Reads token cookie: request.cookies.get("token")?.value
- If absent, checks Authorization header: Bearer <token>
- Public routes allowed:
  - / and auth pages
  - /about, /contact, /pricing, /privacy, /terms
  - /blog
- Special bypass: paths starting with:
  - /api/webhook
  - /api/public
  - /api/auth

If route is protected and no token:
- redirects to /auth/login?from=<originalPath>

Loop prevention:
- If token cookie exists but is expired, middleware does not always redirect away from auth pages to avoid redirect loops.
- AuthProvider handles post-login redirect UX.

5.2 API-level auth: src/app/lib/middleware.js (withAuth)
withAuth(handler, options)
- Extracts token from:
  - cookie header parsing (cookieHeader -> cookiesMap.token)
  - Authorization header (Bearer)
  - next/headers() fallback
- Verifies token via verifyToken from src/app/lib/auth.js
- Loads and validates subscription/user status via validateSubscriptionStatus from planMiddleware.js
- Enforces:
  - account active (user.status === "active")
  - account not locked (user.isLocked)
- Sets req.user = user
- Supports:
  - options.optional: allow handler to run without user
  - options.roles: role-based authorization

5.3 CSRF protection: src/app/lib/withCsrf.js and/or withCsrf export in same lib file
withCsrf(handler)
- For safe methods: passes through (GET/HEAD/OPTIONS)
- For unsafe methods:
  - reads CSRF header name (CSRF_HEADER_NAME)
  - reads CSRF cookie name (CSRF_COOKIE_NAME)
  - validateCsrfToken(csrfHeader, csrfCookie)
- If invalid: returns 403 with CSRF_VALIDATION_FAILED

5.4 Error handling: withErrorHandling(handler)
- Wraps handler execution
- Catches exceptions and returns standardized JSON:
  - error: "Internal server error"
  - message: detailed in development, generic in production
  - code, status, optional stack in development

6. Plans, Authorization, and Limits
============================================================
6.1 Tier constants
In src/app/lib/planMiddleware.js:
- TIERS: free, pro, enterprise
- TIER_LEVELS: free=0, pro=1, enterprise=2

6.2 Subscription validity & auto-downgrade
validateSubscriptionStatus(userId)
- Loads user from MongoDB users collection
- Determines effective expiry from:
  - subscription.currentPeriodEnd || expiryDate || expiresAt
- If expired and tier is not free:
  - updates user subscription fields to FREE
  - sets status "expired", timestamps, clears expiry dates
  - sets isPremium=false
- Returns downgraded user object or null on errors

6.3 Plan quota enforcement
Rate limiting/quotas middleware: withAPIRateLimit(handler, apiName)
- Computes limit via checkAPILimit(user._id, apiName)
- If exceeded:
  - returns 429 JSON with:
    - limit, used, remaining, nextReset
- Explicit usage increments are NOT automatic here.
  - Handlers must call trackAPIUsage() after successful expensive operations.

6.4 Usage tracking
trackAPIUsage(userId, apiName)
- Maps API name to plan feature key via getFeatureName:
  - generate-course -> generateCourseLimit
  - generate-flashcards -> flashcards
  - quiz -> quizGenerations
  - ai-tutor-chat -> aiResponses
  - report outline/section -> reportGenerations
  - career routes -> careerLimit
- Increments api_usage collection with:
  - userId
  - apiName(featureName)
  - month bucket (monthStart = first day)

6.5 Course access control & sharing
checkCourseAccess(userId, courseId, shareId)
- If shareId provided:
  - checks library collection for matching share configs or shared course record
  - determines sharer tier (from sharePlan/isPremium or shareConfigs matching shareId)
  - paid sharer -> fullAccess=true
  - free sharer -> limited access (fullAccess=false)
- If no shareId:
  - validates subscription first (auto-downgrade)
  - checks if course is in library:
    - owner access always full
    - enrollees may receive full access depending on whether access came via a shared chain
  - if course is not in library:
    - if course is not premium -> public access
    - if premium -> requires paid plan unless enrolled
    - tierRequired and TIER_LEVELS determine access

8. Core Feature Documentation (Key Flows)
============================================================
8.1 Course generation
Endpoint: POST /api/generate-course

Security:
- requires auth + CSRF

Behavior:
- validates topic/difficulty/format
- enforces monthly quota via planMiddleware
- canonicalizes topic to detect duplicates
- reuses existing course if already generated
- creates structured course outline (commonly 20 modules / 100 lessons)
- stores course in library
- free users are restricted to early modules

8.2 Flashcards generation
Endpoint: POST /api/generate-flashcards

Behavior:
- generates card sets with SRS metadata
- stores in cardSets
- supports appending to existing set
- avoids duplicate sets for same topic/difficulty

8.3 Quizzes generation and practice
Behavior:
- quizzes are generated under course-gen workflow with format=quiz
- stored in tests collection
- supports quiz reuse for same topic/difficulty
- performance tracked in quizzes/[id]/performance

8.4 Reports / essays
Endpoints:
- POST /api/generate-report-outline
- POST /api/generate-report-section
- /api/reports management

Behavior:
- outline creates section metadata first
- section generation produces paragraph arrays plus references
- report editor workflow stores aggregate document state
- report generation is Pro-only (enforced by plan limits)

8.5 AI tutor chat
Endpoint: POST /api/chat

Behavior:
- requires auth + CSRF
- topic-scoped tutoring only (focused learning)
- validates message/topic/history array shape
- uses plan enforcement via withAPIRateLimit
- increments usage via trackAPIUsage after success

8.6 Tutor chat history
Endpoints (example): src/app/api/chat/history/route.js
- GET/POST/DELETE guarded by withAuth
