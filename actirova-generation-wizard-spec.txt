-------------------------------------------------------------------
SECTION A — UNIVERSAL WIZARD STRUCTURE (5 steps, same shape as your
current screenshots: Journal Type -> ... -> ... -> ... -> Review)
-------------------------------------------------------------------

STEP 1 — PAPER TYPE
  Card grid, single-select. (Already built — 18 cards, one per type.)

STEP 2 — CORE BRIEF  (universal sub-fields + type-specific sub-fields)
  Universal fields present on every type:
    - Topic / Subject            [text input, required, max 200 chars]
        placeholder: "What is this paper about?"
    - Field / Discipline          [text input or dropdown, required]
        placeholder: "e.g. Marketing, Nursing, Computer Science, Law"
    - Academic / Professional Level   [dropdown, required]
        options: High School | Undergraduate | Graduate/Master's |
                 Doctoral | Professional/Business
  Type-specific fields: listed per type in Section C below.

STEP 3 — STRUCTURE & APPROACH  (fully type-specific)
  Fields listed per type in Section C below. Covers methodology,
  angle, framework, options, financial assumptions, etc.

STEP 4 — SOURCES & REQUIREMENTS  (universal fields + type-specific)
  Universal fields present on every type:
    - Sources / Materials Provided   [textarea, optional, max 5000 chars]
        placeholder: "Paste any sources, data, readings, or reference
        materials you want used. Leave blank if none."
    - Citation Style                 [dropdown, required]
        options: APA 7 | MLA 9 | Harvard | Chicago | IEEE | Vancouver |
                 Not Applicable
    - Additional Instructions / Constraints  [textarea, optional, max 3000 chars]
        placeholder: "Paste marking criteria, must-include points,
        tone preferences, or anything else the writer should know."
  Type-specific fields (where noted): listed per type in Section C.

STEP 5 — REVIEW  (universal, same for every type)
    - Page Count       [number input, required, user-entered — no
        type-locked min/max; the user decides how long the paper is]
        placeholder: "How many pages?"
        (optional: show a soft suggested range per type as helper text
        only, e.g. "Most Research Projects run 20-70 pages" — this is
        guidance, not a limit; user can enter any value)
    - Word Count (auto)     [read-only display]
        = Page Count x 500 (academic) or x 550 (business/professional
        types — flag per type below)
    - Credit Cost (auto)    [read-only display]
        = Page Count x 2.5 credits  (see Section B)
    - Summary panel        [read-only, echoes every field from Steps 2-4]
    - [Generate] button    -> submits assembled prompt to AI

-------------------------------------------------------------------
SECTION B — CREDIT SYSTEM
-------------------------------------------------------------------
  Base rate:        50 credits = $10.00  ->  1 credit = $0.20
  Cost per page:     2.5 credits/page  ->  $0.50/page
  Formula:           total_credits = page_count x 2.5
                     total_usd     = total_credits x 0.20

  There is no fixed page range or credit table per type. The user
  types in the exact page count they want on the Review step, for
  ANY paper type, and cost is calculated live from that single input:

      credits = page_count x 2.5
      usd     = credits x 0.20

  Example live-calculation reference (applies to every type, since the
  formula is type-agnostic — only the page count entered changes it):

  Pages entered     Credits     USD
  ----------------- ----------- ----------
  2                  5.0         $1.00
  5                  12.5        $2.50
  10                 25.0        $5.00
  20                 50.0        $10.00
  40                 100.0       $20.00
  70                 175.0       $35.00
  100                250.0       $50.00
  150                375.0       $75.00

  Round displayed credits to nearest 0.5 for the user-facing UI if you
  don't want to show fractional credits (e.g. 62.5 -> 62.5 or 63).

  Optional UX nicety: show a small greyed-out "typical range" hint
  under the Page Count field per type (e.g. "Most users pick 20-70
  pages for this type") purely as a suggestion — it should not block
  or clamp what the user enters.

SECTION D — MASTER ASSEMBLY LOGIC (pseudo-code for your backend)
-------------------------------------------------------------------

  function buildFinalPrompt(paperType, step2Fields, step3Fields,
                             step4Fields, pageCount):

      wordsPerPage = (paperType in ["Business Report","Grant Proposal",
                       "Business Plan","Policy Brief","Project Proposal"])
                       ? 550 : 500
      wordCount   = pageCount * wordsPerPage
      credits     = pageCount * 2.5
      usd         = credits * 0.20

      prompt = SHARED_HUMANIZATION_LAYER
                 .replace("{DOCUMENT_TYPE}", paperType)
                 .replace("{PAGE_RANGE}", pageCount)
                 .replace("{WORD_RANGE}", wordCount)

      prompt += SYSTEM_PROMPT[paperType]
                 .replace(each "{VARIABLE}" with matching field value
                          from step2Fields / step3Fields / step4Fields)

      if step4Fields.sources is not empty:
          prompt += "\nUse ONLY these provided sources for citations:\n"
                     + step4Fields.sources
      else if paperType requires real citations (Annotated Bibliography,
              Dissertation, Literature Review, Research Project/Proposal):
          prompt += "\nNo sources were provided. Do not fabricate
                     citations, statistics, or quotes — flag this
                     limitation instead."

      prompt += "\nCitation style: " + step4Fields.citationStyle
      prompt += "\nAdditional requirements: " + step4Fields.instructions

      return {prompt, wordCount, credits, usd}

  On [Generate] click:
      1. Validate all required fields are filled.
      2. Compute credits via formula above; show final confirm modal
         ("This will cost X credits (~$Y). Continue?").
      3. Deduct credits from user balance only after successful AI
         response (or hold/escrow at submit, refund on failure).
      4. Submit assembled prompt to generation model.
      5. Stream/display output; log actual word count vs. target for
         QA (flag if output misses target by >15%).
